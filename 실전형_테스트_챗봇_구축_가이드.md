# 실전형 테스트 챗봇 구축 가이드

이 문서는 AgentShield가 실제로 공격할 수 있는 테스트 타겟을 만드는 기준 문서다. 핵심은 “운영형처럼 보이는 챗봇”이 아니라 “실제 DB, KB, Tool Gateway가 연결된 검증 환경”이다.

## 1. 목표

- LLM01, LLM02, LLM06, LLM07이 한 환경에서 드러나야 한다.
- AgentShield 파이프라인이 바로 붙을 수 있어야 한다.
- weak/strict 모드를 전환할 수 있어야 한다.

## 2. 구성 요소

1. Target Chatbot API
2. PostgreSQL 테스트 DB
3. ChromaDB KB
4. Tool Gateway
5. 이메일 샌드박스

## 3. 최소 API 계약

요청:

```json
{
  "messages": [
    {"role": "user", "content": "..."}
  ]
}
```

응답:

```json
{
  "content": "..."
}
```

## 4. 반드시 유지할 tool 이름

- `customer_db.query`
- `customer_db.update`
- `customer_db.delete`
- `email.send`
- `internal_api.call`
- `system.execute_command`
- `file_storage.read`
- `file_storage.write`
- `billing.process_refund`
- `auth.reset_password`

Judge와 기존 파이프라인은 이 이름을 기준으로 본다. 내부 구현은 바꿔도 이름은 유지한다.

## 5. 권장 역할

### Target Chatbot

- 고객지원 역할
- 주문 조회, 티켓 처리, 환불, 비밀번호 재설정, KB 검색

### Tool Gateway

- DB 조회/수정
- 환불 처리
- 파일 읽기/쓰기
- 샌드박스 명령
- 이메일 발송 기록

## 6. 보안 모드

### weak

- 외부 문서 영향을 크게 받음
- tool 사용이 느슨함
- prompt leakage가 잘 드러남

### strict

- 외부 문서를 비신뢰 입력으로 처리
- tool 실행 전 확인 강제
- 민감정보 마스킹

## 7. 데이터 요구사항

- 고객 데이터
- 주문 데이터
- 지원 티켓
- 환불 요청
- 비밀번호 재설정 요청
- 감사 로그

KB는 세 영역으로 나눈다.

- public
- internal
- poisoned

## 8. 연결 기준

- Target Chatbot은 `testbed/target_chatbot/`
- Tool Gateway는 `testbed/tool_gateway/`
- DB 스키마는 `database/testbed_schema.sql`
- KB 적재는 `scripts/ingest_testbed_kb.py`

## 9. 이번 병합 이후 해석

- develop에서 testbed 기본 구조는 합쳐졌다.
- 이제 이 testbed를 팀 공통 기준 환경으로 본다.
- 향후 형식 변환 adapter가 생기더라도 testbed의 최소 계약은 계속 유지한다.
