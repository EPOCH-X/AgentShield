AgentShield 대시보드 — 목업(API 폴백) 흐름별 사용법
=====================================================

전제
----
1. dashboard 폴더에서 개발 서버 실행:
   npm run dev
   (또는 캐시 꼬임 시: npm run dev:fresh)

2. 목업은 "백엔드(FastAPI)에 연결하지 못했을 때" 자동으로 동작합니다.
   - 기본 백엔드 주소: http://127.0.0.1:8000
   - 다른 주소를 쓰려면 환경 변수: AGENTSHIELD_API_URL

3. 목 응답에는 HTTP 헤더 x-agentshield-mock: 1 이 붙습니다. (브라우저 개발자 도구 Network 탭에서 확인)

4. 스캔 목 세션 ID(고정): mock-session-demo
   스캔 시작(POST) 응답으로 이 ID가 오므로, 상세·보고서 URL이 이 값과 맞춰집니다.


흐름 1 — 로그인 (/login)
------------------------
목적: JWT 없이 콘솔(스캔 등) 화면 진입 연습

사용법:
1. 백엔드를 끈 상태에서 http://localhost:3000/login 접속
2. 아이디·비밀번호에 각각 한 글자 이상 입력 후 로그인
3. 성공 시 로컬 스토리지에 access_token 저장 → /scan 으로 이동

비고:
- 빈 아이디/비밀번호는 목에서 422
- 백엔드가 켜져 있으면 실제 /api/v1/auth/login 이 호출됨


흐름 2 — 회원가입 (/signup)
---------------------------
목적: 가입 폼·성공 UI 확인

사용법:
1. 백엔드 끈 상태에서 /signup
2. 이메일·아이디·비밀번호·비밀번호 확인 입력 (비밀번호 8자 이상)
3. 제출 시 목 메시지: "가입이 완료되었습니다. (목업)"

비고:
- 비밀번호 8자 미만은 목에서 422
- 백엔드 ON 시 실제 등록 API 호출


흐름 3 — LLM 스캔 시작 (/scan)
--------------------------------
목적: 스캔 세션 생성 후 상세 페이지로 이동

사용법:
1. 로그인 완료 후 /scan (또는 홈 리다이렉트)
2. 프로젝트명·대상 URL 입력 → "시스템 스캔 시작"
3. 목 응답: session_id = mock-session-demo, status = running
4. 자동으로 /scan/mock-session-demo 로 이동

비고:
- 공격 벡터 토글은 현재 API 본문에 포함되지 않음(화면 안내용)


흐름 4 — 스캔 상세 (/scan/[id])
-------------------------------
목적: Phase·로그·취약점 목록 폴링

사용법:
1. [id] 가 mock-session-demo 이거나 임의 문자열이어도 목 동작
2. GET .../scan/{id}/status → completed, phase 4, 통계 채움
3. GET .../scan/{id}/results → 샘플 취약점 5건 (phase·category·severity·defense_code 등)
4. 쿼리: ?category=LLM01 &severity=high &phase=1 형태로 필터 가능(목 데이터 기준)

비고:
- 백엔드 ON이면 실 DB/파이프라인 결과


흐름 5 — 보고서 PDF (/report/[id])
----------------------------------
목적: PDF 다운로드 버튼·Blob 저장

사용법:
1. /report/mock-session-demo (또는 임의 세션 id)
2. "PDF 다운로드" 클릭
3. GET .../report/{id}/pdf → application/pdf 목 파일

비고:
- 뷰어에 따라 PDF 경고가 날 수 있음(목 구조 한계). 개발용 확인용.


흐름 6 — 모니터링 및 위반 (/monitoring)
-----------------------------------------
목적: 대시보드 카드·위반 테이블·필터

사용법:
1. 로그인 후 사이드바 "모니터링 및 위반"
2. GET .../monitoring/dashboard → 일일 요청·위반 수 등
3. GET .../monitoring/violations → 샘플 위반 5건
4. 부서·위반 유형 필터 후 "필터 적용" → 쿼리 반영 목

비고:
- 로드 실패 배너는 네트워크/401 등; 목 성공 시에는 안 뜸


흐름 7 — 관리자 (/monitoring/admin)
------------------------------------
목적: 정책 목록·직원 목록·정책 추가

사용법:
1. "관리자" 메뉴
2. GET .../monitoring/employees → 직원 5명(목)
3. GET .../monitoring/policies → 정책 3건(목) + 이전에 추가한 항목은 서버 프로세스 메모리에 누적(개발 서버 재시작 시 초기화)
4. 모달에서 정책 작성 후 저장 → POST .../monitoring/policies 목이 새 행 반환

비고:
- 실제 운영에서는 관리자 권한·백엔드 검증이 필요할 수 있음


흐름 8 — 플랫폼 개요 (/overview)
---------------------------------
목적: 기능 A/B 설명·공유 컴포넌트 표 (API 없음)

사용법:
1. 사이드바 "플랫폼 개요" 클릭
2. 정적 콘텐츠만 표시 → 백엔드 불필요

문제 시
-------
- ECONNREFUSED 로그가 계속이면: 이전 rewrite 대신 API 라우트 폴백이 적용됐는지 next.config.js 확인
- 청크 오류(682.js 등): npm run clean 후 npm run dev 또는 npm run dev:fresh
- 목이 아니라 503만 온다: 해당 경로는 buildMockResponse 에 없음 → lib/devBackendMock.ts 에 핸들러 추가
