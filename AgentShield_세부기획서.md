# AgentShield 세부기획서

이 문서는 팀이 같은 결과물을 만들기 위한 기준 문서다. 구현 아이디어를 길게 적는 문서가 아니라, 역할, 산출물, 인터페이스, 운영 원칙을 고정하는 문서로 본다.

## 1. 프로젝트 목표

- 기능 A: AI 챗봇 보안 테스트 자동화
- 기능 B: 직원 AI 사용 모니터링
- 공통 목표: 공격, 방어, 검증, 운영 로그를 하나의 제품 구조로 연결

## 2. 현재 기준 산출물

### 기능 A

- Phase 1 스캐너
- Phase 2 Red Agent
- Judge
- Phase 4 Verify
- 결과 저장 및 요약

### 기능 B

- Monitoring Proxy 요청 흐름
- 정책 탐지
- 의도 재검토
- 사용 로그/위반 기록 연결 지점

### 공통

- PostgreSQL
- ChromaDB
- Ollama 기반 모델 호출
- FastAPI 백엔드
- Next.js 대시보드
- Testbed

## 3. 역할 분담 원칙

- 역할은 유지하되 문서와 인터페이스는 통일한다.
- 한 사람이 맡은 폴더라도 계약은 팀 전체 기준으로 고정한다.
- 공통 규약을 바꿀 때는 README와 파이프라인 문서가 같이 바뀌어야 한다.

## 5. 반드시 고정할 인터페이스

### 타겟 챗봇 최소 계약

요청:

```json
{
  "messages": [
    {"role": "user", "content": "..."}
  ]
}
```

응답:

```json
{
  "content": "..."
}
```

### 현재 한계

- 이 계약이 여러 파일에 직접 박혀 있다.
- 고객 친화 통합을 위해서는 공통 adapter 계층이 필요하다.

## 7. 산출물 기준

### 구현 완료로 보는 조건

- 코드가 있다.
- 호출 경로가 연결돼 있다.
- 최소 실행 절차가 문서화돼 있다.
- 테스트 또는 수동 검증 기준이 있다.

### 미완으로 보는 조건

- 함수/엔드포인트가 placeholder다.
- 로컬 데모만 되고 API 경로가 연결되지 않는다.
- 문서가 목표만 말하고 현재 상태를 숨긴다.

## 8. 문서 운영 원칙

- README는 입구 역할만 한다.
- 개요 문서는 제품 설명만 맡는다.
- 파이프라인 문서는 구현 계약만 맡는다.
- QA 문서는 실행 절차만 맡는다.
- testbed 문서는 테스트 타겟 설계와 실행만 맡는다.

## 9. 이번 기준에서 팀이 맞춰야 할 것

- tool 이름은 함부로 바꾸지 않는다.
- judge 계약을 조용히 바꾸지 않는다.
- 개인 메모성 분석 문서는 공식 기준 문서에 섞지 않는다.
- 결과 요약은 숫자, 범위, 해석만 남기고 장황한 서술은 뺀다.
